# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано
- Использовался синтетический датасет `S12-hw-dataset.csv` для прогнозирования числового значения `target` на один шаг вперед.
- Организован строгий хронологический `train / validation / test` сплит для предотвращения утечек данных из будущего (data leakage).
- Сравнивались четыре подхода: наивный прогноз (B1), скользящее среднее (B2), Ridge-регрессия на лаговых признаках (B3) и рекуррентная нейросеть GRU (R1).
- На этапе валидации лучшим подходом оказалась Ridge-регрессия (B3), так как она идеально уловила явно заданную недельную сезонность через сгенерированные признаки. Тем не менее, для финального тестирования пайплайна была оценена модель GRU (R1).

## 2. Среда и воспроизводимость
- Python: 3.10+
- pandas / numpy / scikit-learn: Использованы актуальные версии.
- torch: 2.x
- Устройство (CPU/GPU): CUDA
- Seed: 42
- Как запустить: открыть `HW12.ipynb` и выполнить Run All.

## 3. Данные и постановка задачи
- Датасет: `S12-hw-dataset.csv`
- Целевая колонка: `target`
- Частота временного ряда: Ежедневная (Daily)
- Горизонт прогноза: 1 шаг вперед
- Размер датасета: 1000 строк
- Диапазон дат: 2024-01-01 — 2026-09-26
- Есть ли дополнительные внешние признаки: Нет, использован только сам исторический ряд `target`.
- Комментарий: В ряде присутствует выраженный линейный тренд, наложенная сезонность (недельная и годовая) и нормальный шум. Для такого ряда критически важно не применять случайное перемешивание при разбиении, чтобы модель честно экстраполировала тренд в будущее.

## 4. Temporal split и признаки

### 4.1. Разбиение по времени
- Как именно выполнен `train / validation / test`: Датасет отсортирован по времени, затем разбит по индексам.
- Какие доли или интервалы использовались: 70% (train) / 15% (val) / 15% (test).
- Почему `random split` здесь некорректен: Случайное разбиение отправляет данные из будущего в обучающую выборку. Модель просто "запомнит" будущие значения, что сделает оценку качества прогноза абсолютно нерелевантной.

### 4.2. Признаки для baseline-моделей
- Какие лаги использовались: `lag_1`, `lag_7`, `lag_14`.
- Какие rolling-признаки использовались: `rolling_mean_7`, `rolling_std_7` (посчитанные от `lag_1` для избежания заглядывания в текущий таргет).
- Какие календарные признаки использовались: `dayofweek` (день недели).
- Как обрабатывались пропуски после построения лагов/rolling: Удалялись с помощью `.dropna()`.
- Использовалось ли масштабирование и где именно оно обучалось: Применялся `StandardScaler`. Метод `.fit()` вызывался строго только на обучающей (`train`) выборке, чтобы параметры распределения не утекли из валидации или теста.

## 5. Модели и эксперименты (B1, B2, B3, R1)
Опишите коротко и сопоставимо:
- B1 (`naive-last`): Прогноз равен фактическому значению предыдущего дня.
- B2 (`moving-average`): Прогноз равен среднему значению за последние 7 дней (MA 7).
- B3 (`ridge-lag-features`): Линейная регрессия с L2-регуляризацией (Ridge), обученная на лагах и календарных признаках.
- R1 (`gru-forecast`): Рекуррентная нейросеть GRU (64 скрытых нейрона), обученная на сырых оконных последовательностях.

- Основная метрика выбора лучшей модели: `MAE` на валидации.
- Дополнительные метрики: `RMSE`, `MAPE`.
- `window_size` для `GRU`: 14
- Batch size: 32
- Максимальное число эпох: 50
- Optimizer: Adam
- Learning rate: 1e-3
- Как сохранялась лучшая модель: Отслеживался Validation Loss (MSE) в цикле обучения, сохранялся `state_dict` эпохи с минимальной ошибкой.

## 7. Анализ
Наивный бейзлайн (B1) оказался surprisingly сильным по метрике MAE (2.37), так как на ежедневных данных значения меняются плавно, но он принципиально не способен предсказывать развороты тренда. Moving Average (B2) провалился с MAE 8.67, потому что сильное сглаживание за 7 дней заставляет прогноз критически запаздывать за растущим трендом. Ridge (B3) сработала блестяще (MAE 2.31), так как мы вручную передали ей идеальные признаки (`lag_7` и `dayofweek`), которые в точности описывают заложенную в синтетический ряд недельную сезонность. GRU (R1) выучила структуру ряда сама внутри окна `window_size=14`, но уступила классическому ML: для рекуррентных сетей тысяча точек — это слишком мало для самостоятельного извлечения жестких сезонных правил. В подобных задачах главная угроза утечки данных (leakage) возникает при расчете оконных статистик (например, вычисление `rolling_mean` напрямую от таргета без `.shift(1)`). Выбор лучшей модели именно по валидации критичен: это оставляет нам `test` как абсолютно независимую "слепую" выборку для оценки реальной предсказательной способности в продакшене.

## 8. Итоговый вывод
Для старт-апа задачи прогнозирования временного ряда классическая регрессия на грамотно сконструированных лаговых признаках является мощнейшим бейзлайном, который часто бьет сложные модели. Простых подходов (Ridge/LinearRegression) достаточно, когда ряд короткий, а сезонность жестко фиксирована (например, четкая зависимость от дня недели). Переходить к рекуррентным нейросетям (GRU/LSTM) имеет смысл при наличии огромных объемов данных и множества нелинейных экзогенных переменных. Главный урок корректной валидации: время течет в одном направлении, поэтому случайные сплиты строго запрещены, а любые преобразования (`StandardScaler`) должны "видеть" только прошлое.
